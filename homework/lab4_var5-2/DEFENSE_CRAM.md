## 1. Какую задачу решает Ахо--Корасик?

Алгоритм Ахо--Корасик решает задачу поиска множества образцов в одном тексте.

Есть набор образцов:

```text
P = {p1, p2, ..., pk}
```

и текст `T`. Нужно найти все вхождения всех образцов в тексте.

Главная идея: все образцы объединяются в один trie, затем trie превращается в автомат. После этого текст
читается один раз слева направо, и автомат сообщает, какие образцы заканчиваются в текущей позиции.

Асимптотика:

```text
preprocessing: O(sum |pi|)
search:        O(|T| + Ans)
```

где `Ans` -- число найденных вхождений.

В твоей лабораторной символы алфавита -- не `char`, а числа `uint32_t`.

## 2. Какие структуры лежат в основе автомата?

Две ключевые структуры:

1. **Trie**, или дерево ключей.
   - В нём все образцы хранятся как пути из корня.
   - Общие префиксы образцов хранятся один раз.

2. **Связи автомата поверх trie**:
   - `failure_link` -- связь неудачи;
   - `output_link` -- связь выхода.

Можно сказать так:

> Основа автомата -- trie по всем образцам, дополненный связями неудачи и выходными ссылками.

В твоём коде это структура `Node`:

```cpp
struct Node {
    std::unordered_map<std::uint32_t, std::size_t> next;
    std::size_t failure_link = 0;
    int output_link = -1;
    std::vector<PatternInfo> patterns;
};
```

## 3. Trie и наивное использование trie

Trie строится по всем образцам.

Пример образцов:

```text
1 2
1 2 3
1 4
```

В trie общий префикс `1` хранится один раз.

Наивный поиск через trie:

```text
для каждой позиции i в тексте:
    стартуем из корня trie
    идём вправо по тексту, пока есть переход
    если встретили terminal-вершину, нашли образец
```

Проблема: если перехода нет, мы начинаем заново со следующей позиции текста.

Сложность:

```text
O(|T| * L)
```

где `L` -- максимальная длина образца.

В худшем случае это плохо. Например, много совпадающих префиксов заставляют многократно читать одни и
те же символы текста.

Ахо--Корасик ускоряет этот подход за счёт `failure_link`: при неудаче он не начинает с нуля, а переходит
к наибольшему суффиксу текущей строки, который всё ещё может быть префиксом какого-то образца.

## 4. Связь неудачи `failure_link`

`failure_link` из вершины ведёт в вершину, которая соответствует наибольшему собственному суффиксу
текущей строки, являющемуся префиксом какого-то образца.

Пример:

```text
текущая вершина соответствует: 1 2 3 4
failure_link может вести в:     2 3 4
```

если `2 3 4` есть в trie как префикс.

Если подходящего суффикса нет, ссылка ведёт в корень.

В твоём коде построение:

```cpp
while (fallback != 0 && nodes_[fallback].next.find(value) == nodes_[fallback].next.end())
    fallback = nodes_[fallback].failure_link;
```

Смысл:

> Пока из текущего fallback-состояния нельзя перейти по `value`, поднимаемся по связям неудачи.

Почему похоже на КМП:

- в КМП prefix-function говорит, на какой префикс образца откатиться после несовпадения;
- в Ахо--Корасике `failure_link` говорит, в какую вершину trie откатиться после несовпадения;
- оба механизма сохраняют уже прочитанную полезную информацию и не возвращают указатель текста назад.

Препроцессинг:

```text
O(sum |pi|)
```

если переходы хранятся разреженно и каждый trie-edge обрабатывается один раз.

Поиск после этого:

```text
O(|T| + Ans)
```

## 5. Выходная ссылка `output_link`

`output_link` нужна, чтобы находить образцы, которые являются суффиксами текущего состояния.

Пример образцов:

```text
1 2 3
2 3
3
```

Если текст заканчивается на `1 2 3`, то в этой позиции заканчиваются сразу три образца:

```text
1 2 3
2 3
3
```

Если проверять только текущую вершину `1 2 3`, то можно пропустить `2 3` и `3`.

Без `output_link` есть два варианта:

1. **Идти по всей цепочке `failure_link` в каждой позиции.**
   - Корректно, но может быть медленнее.

2. **Проверять только текущую вершину.**
   - Быстро, но некорректно, если один образец является суффиксом другого.

`output_link` решает проблему: она ведёт к ближайшей вершине по цепочке неудач, где заканчивается образец.

В твоём коде:

```cpp
if (!nodes_[suffix].patterns.empty())
    nodes_[child].output_link = static_cast<int>(suffix);
else
    nodes_[child].output_link = nodes_[suffix].output_link;
```

## 6. Поиск с джокером

Типовая задача: образец содержит wildcard, например:

```text
ab?cd??ef
```

`?` совпадает с любым символом.

Подход:

1. Разбить образец на непрерывные блоки без джокеров.

```text
ab?cd??ef
```

даёт блоки:

```text
ab at offset 0
cd at offset 3
ef at offset 7
```

2. Запустить Ахо--Корасик по этим блокам.
3. Когда блок найден в тексте, вычислить возможное начало полного образца:

```text
start = block_start - block_offset
```

4. Для каждого `start` считать, сколько блоков совпало на своих местах.
5. Полный образец найден, если для `start` совпали все блоки.

Если образец состоит только из джокеров длины `m`, то подходят все позиции:

```text
0..|T|-m
```

Доказательство корректности:

- Если полный образец совпал, то каждый его обычный блок обязательно совпал в тексте на своём offset.
- Если все обычные блоки совпали на своих offset, а остальные позиции -- джокеры, то весь образец совпал.

## 7. Упражнения 12, 14, 16

### Упражнение 12

Нужно доказать:

- если никакой образец не является собственной подстрокой другого, поиск идёт за `O(m)`;
- иначе за `O(m + k)`, где `k` -- число вхождений.

Обозначения:

- `m` -- длина текста;
- `k` -- число найденных вхождений.

Ответ:

> Сам проход автомата по тексту занимает `O(m)`, потому что каждый символ текста обрабатывается один раз,
> а переходы по связям неудачи амортизируются. Дополнительная работа возникает только при выводе найденных
> образцов.

Если никакой образец не является собственной подстрокой другого, то в одной позиции текста не может
заканчиваться длинная цепочка вложенных образцов. Значит, число дополнительных выходов ограничено
линейно, и поисковая фаза остаётся `O(m)`.

В общем случае несколько образцов могут заканчиваться в одной позиции. Тогда их нужно вывести, и это
неизбежно стоит `O(k)`. Поэтому итог:

```text
O(m + k)
```

### Упражнение 14

Нужно построить пример, где число вхождений `k` растёт быстрее, чем `O(n + m)`.

Хороший пример:

```text
T = a^N
P = {a, aa, aaa, ..., a^r}
```

Суммарная длина образцов:

```text
M = 1 + 2 + ... + r = Θ(r^2)
```

Число вхождений:

```text
k = N + (N - 1) + ... + (N - r + 1)
  = Θ(Nr)
```

Так как `r = Θ(sqrt(M))`, получаем:

```text
k = Θ(N sqrt(M))
```

Если взять `N` порядка `M`, то:

```text
k = Θ(M^(3/2))
```

Это быстрее линейного `O(N + M)`.

Почему это почти максимум:

в одной позиции может закончиться много образцов только если их длины различны и они вложены как суффиксы.
Чтобы в одной позиции заканчивалось `q` разных образцов, их суммарная длина хотя бы:

```text
1 + 2 + ... + q = Θ(q^2)
```

значит `q = O(sqrt(M))`, а всего позиций `N`, поэтому:

```text
k = O(N sqrt(M))
```

Пример выше достигает этой оценки.

### Упражнение 16

Смысл: не надо анализировать путь отдельно для каждого образца, потому что это завышает время.

В keyword tree `K` каждая вершина и каждое ребро принадлежат общему trie. Общие префиксы образцов не
дублируются.

Более точный анализ:

1. Trie имеет `|V|` вершин и `|E|` рёбер.
2. В дереве `|E| = |V| - 1`.
3. При BFS-построении связей каждая вершина извлекается из очереди один раз.
4. Каждое ребро trie рассматривается один раз.
5. `output_link` вычисляется за `O(1)` из уже построенной связи неудачи, потому что BFS гарантирует,
   что нужная более короткая вершина уже обработана.

Итого препроцессинг связан не с суммой путей отдельно для каждого образца, а с размером самого дерева:

```text
O(|V| + |E|) = O(|V|)
```

Так как число вершин trie не превосходит суммарную длину всех образцов плюс корень:

```text
O(|V|) = O(sum |pi|)
```

## 8. Как рассказывать свой `main.cpp`

Короткая речь:

> Я сначала читаю образцы до пустой строки. Каждый образец парсится как последовательность `uint32_t`,
> поэтому `0002` и `2` становятся одним числом. Затем добавляю все образцы в trie методом `AddPattern`.
> После этого BFS-ом строю связи неудачи и выходные ссылки в `BuildLinks`.
>
> Текст читается построчно, но автомат не сбрасывается между строками, потому что текст -- это один поток
> чисел. Для каждого числа делаю `NextState`, затем через `ForEachMatch` вывожу все образцы, которые
> закончились в этой позиции. Так как автомат знает конец вхождения, а вывести надо начало, я храню очередь
> последних `max_pattern_length` позиций и беру позицию на `pattern.length` назад.

Частые уточнения:

- Почему `unordered_map`, а не массив?
  - Алфавит размера `2^32`, массив невозможен.

- Почему `uint64_t` в парсере?
  - Чтобы безопасно накопить десятичные цифры до `4294967295`, потом привести к `uint32_t`.

- Почему порядок вывода отличается от примера?
  - В условии порядок несущественен.

- Почему строка текста не сбрасывает автомат?
  - Перевод строки нужен только для позиции; сам текст -- последовательность чисел.

## 9. Код 1: наивный поиск подстроки

```cpp
// Код 1: наивный поиск подстроки
#include <iostream>
#include <string>
#include <vector>

std::vector<int> NaiveSearch(const std::string& text, const std::string& pattern) {
    std::vector<int> answer;
    if (pattern.size() > text.size())
        return answer;
    for (std::size_t i = 0; i + pattern.size() <= text.size(); ++i) {
        bool ok = true;
        for (std::size_t j = 0; j < pattern.size(); ++j)
            if (text[i + j] != pattern[j]) {
                ok = false;
                break;
            }
        if (ok)
            answer.push_back(i);
    }
    return answer;
}
```

Сложность:

```text
O(nm)
```

## 10. Код 2: Z-функция за O(n)

`z[i]` -- длина наибольшего префикса строки, совпадающего с подстрокой, начинающейся в `i`.

```cpp
// Код 2: Z-функция за O(n)
#include <algorithm>
#include <iostream>
#include <string>
#include <vector>

std::vector<int> ZFunction(const std::string& s) {
    std::vector<int> z(s.size(), 0);
    int left = 0;
    int right = 0;
    for (int i = 1; i < s.size(); ++i) {
        if (i <= right)
            z[i] = std::min(right - i + 1, z[i - left]);
        while (i + z[i] < s.size() && s[z[i]] == s[i + z[i]])
            ++z[i];
        if (i + z[i] - 1 > right) {
            left = i;
            right = i + z[i] - 1;
        }
    }
    return z;
}
```

## 11. Код 3: построение sp / prefix-function

В стандартной терминологии это prefix-function `pi`.

`pi[i]` -- длина максимального собственного префикса `s[0..i]`, который одновременно является суффиксом
`s[0..i]`.

```cpp
// Код 3: построение sp / prefix-function
#include <iostream>
#include <string>
#include <vector>

std::vector<int> PrefixFunction(const std::string& s) {
    std::vector<int> pi(s.size(), 0);
    for (int i = 1; i < s.size(); ++i) {
        int j = pi[i - 1];
        while (j > 0 && s[i] != s[j])
            j = pi[j - 1];
        if (s[i] == s[j])
            ++j;
        pi[i] = j;
    }
    return pi;
}
```

Сложность:

```text
O(n)
```

## 12. Код 4: построение sp из Z-функции

Идея:

если `z[i] = len`, то префикс длины `len` совпадает с подстрокой `s[i..i+len-1]`. Значит, для всех
позиций конца внутри этого блока можно поставить кандидата на prefix-function.

Надёжный вариант:

```cpp
// Код 4: построение sp из Z-функции
#include <algorithm>
#include <iostream>
#include <string>
#include <vector>

std::vector<int> PrefixFromZ(const std::string& s) {
    std::vector<int> z = ZFunction(s);
    std::vector<int> pi(s.size(), 0);
    for (int i = 1; i < s.size(); ++i)
        for (int j = z[i] - 1; j >= 0; --j) {
            int pos = i + j;
            if (pi[pos] != 0)
                break;
            pi[pos] = j + 1;
        }
    return pi;
}
```

Почему суммарно `O(n)`:

каждая позиция `pi[pos]` присваивается не более одного раза; когда встречаем уже заполненную позицию,
останавливаем внутренний цикл.

## 13. Код 5: правило плохого символа в Бойере--Муре

Для каждого символа храним его последнее вхождение в образце.

```cpp
// Код 5: правило плохого символа в Бойере--Муре
#include <algorithm>
#include <iostream>
#include <string>
#include <vector>

std::vector<int> BuildBadChar(const std::string& pattern) {
    const int ALPHABET = 256;
    std::vector<int> last(ALPHABET, -1);
    for (int i = 0; i < pattern.size(); ++i)
        last[pattern[i]] = i;
    return last;
}
```

Использование при поиске:

```cpp
std::vector<int> BoyerMooreBadChar(const std::string& text, const std::string& pattern) {
    std::vector<int> answer;
    std::vector<int> last = BuildBadChar(pattern);

    int n = text.size();
    int m = pattern.size();
    int shift = 0;

    while (shift <= n - m) {
        int j = m - 1;

        while (j >= 0 && pattern[j] == text[shift + j])
            --j;

        if (j < 0) {
            answer.push_back(shift);
            ++shift;
        } else {
            int bad = text[shift + j];
            shift += std::max(1, j - last[bad]);
        }
    }

    return answer;
}
```

## 14. Код 6: trie -- insert/search/delete

```cpp
// Код 6: trie -- insert/search/delete
#include <iostream>
#include <string>
#include <unordered_map>

struct TrieNode {
    std::unordered_map<char, TrieNode*> next;
    bool terminal = false;
};

void Insert(TrieNode* root, const std::string& word) {
    TrieNode* current = root;
    for (char c : word) {
        if (current->next.find(c) == current->next.end())
            current->next[c] = new TrieNode();
        current = current->next[c];
    }
    current->terminal = true;
}

bool Search(TrieNode* root, const std::string& word) {
    TrieNode* current = root;
    for (char c : word) {
        auto found = current->next.find(c);
        if (found == current->next.end())
            return false;
        current = found->second;
    }
    return current->terminal;
}

bool Delete(TrieNode* current, const std::string& word, std::size_t index = 0) {
    if (index == word.size()) {
        if (!current->terminal)
            return false;
        current->terminal = false;
        return current->next.empty();
    }
    char c = word[index];
    auto found = current->next.find(c);
    if (found == current->next.end())
        return false;
    bool should_delete_child = Delete(found->second, word, index + 1);
    if (should_delete_child) {
        delete found->second;
        current->next.erase(found);
    }
    return current->next.empty() && !current->terminal;
}

void Destroy(TrieNode* current) {
    for (auto& edge : current->next)
        Destroy(edge.second);
    delete current;
}
```

Важно: если вызываешь `Delete(root, word)`, сам `root` удалять нельзя. Возвращаемое значение для корня
надо игнорировать.

## 15. Минимум, который надо сказать уверенно

Если времени совсем мало, выучи это:

> Ахо--Корасик строит trie по всем образцам. Потом для каждой вершины строятся связи неудачи: они ведут к
> наибольшему суффиксу текущего пути, который является префиксом какого-то образца. Это как fallback в
> КМП, только не для одного образца, а для всего trie. Выходные ссылки нужны, чтобы найти все образцы,
> которые заканчиваются в текущей позиции, включая суффиксы более длинных образцов. После построения
> автомат читает текст один раз, поэтому поиск работает за `O(T + Ans)`.

Для своего варианта добавь:

> В моей задаче алфавит -- числа `0..2^32-1`, поэтому переходы хранятся как
> `unordered_map<uint32_t, size_t>`, а не как массив по символам.
