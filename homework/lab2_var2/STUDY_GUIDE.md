## Блок 1. Что вообще делает программа

### Что понять

Надо понять задачу с точки зрения пользователя, вообще без погружения в код.

### Что происходит в коде

Программа — это словарь:

- ключ — слово из английских букв;
- регистр не важен;
- значением является `uint64_t`, то есть целое число от `0` до `2^64 - 1`.

Поддерживаются 4 вида команд:

1. `+ word number`
   Добавить слово.

2. `- word`
   Удалить слово.

3. `word`
   Найти слово.

4. `! Save path` / `! Load path`
   Сохранить словарь в файл или загрузить из файла.

Если коротко, программа делает две большие вещи:

- хранит слова в памяти в красно-чёрном дереве;
- умеет сохранять и восстанавливать это дерево через собственный бинарный формат файла.

3. Что должно произойти при `Load`, если файла нет?

## Блок 2. Какие C++-конструкции тут используются

### Что понять

У тебя слабее именно C++, поэтому надо сначала снять технический страх перед синтаксисом.

### Что происходит в коде

Весь код построен вокруг класса:

```cpp
class RedBlackTreeDictionary
```

Это просто "тип данных", внутри которого лежат:

- поля;
- вложенные структуры;
- методы.

Главные C++-элементы здесь:

#### 1. `private` и `public`

- `public` — то, чем можно пользоваться снаружи класса;
- `private` — внутренняя кухня класса.

Снаружи нам доступны:

- конструктор;
- деструктор;
- `Insert`;
- `Erase`;
- `Find`;
- `Save`;
- `Load`;
- `Swap`.

Всё остальное — это служебные детали.

#### 2. `struct Node`

Это описание одного узла дерева.

В нём лежат:

- `key`
- `value`
- `color`
- `parent`
- `left`
- `right`

То есть каждый узел знает:

- своё слово;
- своё число;
- свой цвет;
- своего родителя;
- левого и правого ребёнка.

#### 3. Конструктор

Например:

```cpp
RedBlackTreeDictionary()
    : root_(nullptr), nil_(new Node()), size_(0) {
```

Это функция, которая вызывается при создании объекта.

Здесь создаётся пустой словарь:

- `root_` — корень;
- `nil_` — специальный фиктивный узел;
- `size_` — число элементов.

#### 4. Деструктор

```cpp
~RedBlackTreeDictionary()
```

Вызывается при уничтожении объекта.

Здесь он:

- очищает всё дерево;
- удаляет фиктивный узел `nil_`.

#### 5. `static`

Например:

```cpp
static char ToLowerAscii(char symbol)
```

`static`-метод не работает с конкретным объектом словаря.
Он просто служебная функция, которую удобно держать внутри класса.

#### 6. Ссылки и указатели

Здесь очень важно различать:

- `Node*` — указатель на узел;
- `uint64_t& value` — ссылка на переменную.

Пример:

```cpp
bool Find(const std::string& key, uint64_t& value) const
```

Это значит:

- `key` передаётся по константной ссылке, чтобы не копировать строку;
- `value` передаётся по ссылке, чтобы функция могла записать туда найденное число.

#### 7. `const`

Если метод заканчивается на `const`, например:

```cpp
bool Find(...) const
```

это значит, что метод не должен менять состояние словаря.

#### 8. `delete` копирования

```cpp
RedBlackTreeDictionary(const RedBlackTreeDictionary&) = delete;
RedBlackTreeDictionary& operator=(const RedBlackTreeDictionary&) = delete;
```

Это запрет копирования.

Почему это разумно:

- внутри есть сырые указатели;
- если случайно скопировать объект "поверхностно", можно сломать память.

#### 9. `std::move`

Используется, когда строку уже можно "передать дальше", а не копировать.

Пример:

```cpp
Node* inserted = new Node(std::move(key), value, RED, nil_);
```

Смысл:

- `key` уже не нужна как отдельная переменная;
- можно передать её внутрь узла дешевле.

#### 10. `std::bad_alloc`

В `main` есть:

```cpp
catch (const std::bad_alloc&)
```

Это перехват ошибки нехватки памяти.

### Самопроверка

1. Чем `public` отличается от `private`?
2. Чем указатель отличается от ссылки?
3. Зачем методам `Find` и `Save` нужен `const`?
4. Почему здесь запретили копирование класса?
5. Зачем нужен деструктор?

### Переходи дальше, если...

Ты можешь спокойно прочитать сигнатуру функции и объяснить, что означает каждый символ вроде `*`, `&`, `const`.

## Блок 3. Как словарь хранится внутри

### Что понять

Теперь надо понять внутреннее устройство объекта `RedBlackTreeDictionary`.

### Что происходит в коде

У словаря есть 3 главных поля:

```cpp
Node* root_;
Node* nil_;
uint64_t size_;
```

#### 1. `root_`

Это корень дерева.

Если дерево пустое, `root_ == nil_`.

#### 2. `size_`

Количество слов в словаре.

Используется:

- при `Save`, чтобы записать в файл число элементов;
- просто как часть состояния словаря.

#### 3. `nil_`

Это самая важная идея всей реализации.

`nil_` — это специальный фиктивный узел-сентинел.

Вместо `nullptr` код почти везде использует `nil_`.

Почему это удобно:

- у `nil_` есть цвет `BLACK`;
- у него можно безопасно читать поля вроде `color`, `left`, `right`, `parent`;
- это сильно упрощает RB-алгоритмы.

Именно поэтому в конструкторе делается:

```cpp
nil_->color = BLACK;
nil_->parent = nil_;
nil_->left = nil_;
nil_->right = nil_;
root_ = nil_;
```

То есть `nil_` ссылается сам на себя.

Это не ошибка. Это специально.

### Почему `nil_` лучше, чем `nullptr`

Если бы вместо него был `nullptr`, то в `InsertFixup` и `DeleteFixup` пришлось бы постоянно писать дополнительные проверки:

- есть ли брат?
- есть ли дядя?
- есть ли левый ребёнок?
- не `nullptr` ли это?

С `nil_` всё проще:

- если ребёнка нет, это просто `nil_`;
- `nil_` гарантированно чёрный;
- алгоритм получается намного чище.

### Самопроверка

1. Что хранится в одном узле?
2. Почему у словаря есть и `root_`, и `size_`, и `nil_`?
3. Почему пустое дерево — это `root_ == nil_`, а не `root_ == nullptr`?
4. Почему у `nil_` цвет именно чёрный?

## Блок 4. Нормализация слов и поиск

### Что понять

Перед вставкой, удалением и поиском слово нормализуется.

### Что происходит в коде

Есть две похожие вещи:

#### 1. `NormalizeWord`

Находится вне класса.

Она переводит слово в нижний регистр:

```cpp
static std::string NormalizeWord(const std::string& word)
```

Именно она используется в `main`, когда пользователь вводит `A`, `Apple`, `HELLO`.

То есть в дерево всегда попадает уже нормализованное слово.

#### 2. `ToLowerAscii`

Это внутренняя служебная функция класса.

Она тоже переводит символ в нижний регистр, но нужна уже при `Load`, когда мы читаем слово из файла.

### Как работает поиск

Есть внутренняя функция:

```cpp
Node* FindNode(const std::string& key) const
```

Она идёт по обычному BST-правилу:

- если ключ равен текущему — нашли;
- если меньше — идём влево;
- если больше — вправо.

Если дошли до `nil_`, значит слова нет.

Публичная функция `Find` просто:

1. вызывает `FindNode`;
2. если нашли — кладёт значение в `value`;
3. возвращает `true` или `false`.

### `Minimum`

Функция:

```cpp
Node* Minimum(Node* node) const
```

идёт всё время влево, пока может.

Это нужно при удалении:

- когда у удаляемого узла есть оба ребёнка;
- тогда берётся его следующий по порядку элемент, то есть минимум в правом поддереве.

### Самопроверка

1. Зачем вообще нормализовать слово до вставки?
2. Где именно происходит перевод в нижний регистр?
3. Чем `FindNode` отличается от `Find`?
4. Зачем нужна `Minimum`?

### Переходи дальше, если...

Ты можешь сам на бумаге пройти поиск по дереву для 4-5 слов.

## Блок 5. Вставка: сначала как в BST, потом как в RB-дереве

### Что понять

Вставка состоит из двух частей:

1. обычная вставка как в бинарное дерево поиска;
2. восстановление инвариантов красно-чёрного дерева.

### Часть 1. Обычная BST-вставка

Функция:

```cpp
bool Insert(std::string key, uint64_t value)
```

Что она делает:

1. идёт по дереву и ищет место вставки;
2. если такой ключ уже есть — возвращает `false`;
3. создаёт новый узел;
4. прикрепляет его как левого или правого ребёнка;
5. вызывает `InsertFixup`;
6. увеличивает `size_`.

### Почему новый узел красный

Новый узел создаётся так:

```cpp
new Node(std::move(key), value, RED, nil_)
```

То есть новый узел сначала красный.

Почему не чёрный:

- если делать новый узел чёрным, можно сломать чёрную высоту;
- красный узел безопаснее вставлять, а потом уже чинить дерево.

### Часть 2. `InsertFixup`

Это стандартное восстановление RB-инвариантов.

Главная проблема после вставки:

- новый узел красный;
- его родитель тоже может оказаться красным;
- тогда нарушается правило "у красного узла не может быть красного ребёнка".

Пока родитель красный, выполняется цикл.

### Случаи вставки

Разбирать нужно не код построчно, а идею.

#### Случай 1. Дядя красный

Если у текущего узла:

- родитель красный;
- дядя тоже красный;

то делаем перекраску:

- родителя в чёрный;
- дядю в чёрный;
- дедушку в красный.

Проблема поднимается выше, к дедушке.

#### Случай 2. "Треугольник"

Например:

- родитель — левый ребёнок дедушки;
- текущий узел — правый ребёнок родителя.

Тогда сначала делаем поворот у родителя, чтобы превратить треугольник в линию.

#### Случай 3. "Линия"

После этого:

- родитель красится в чёрный;
- дедушка в красный;
- делается поворот у дедушки.

### Почему есть левая и правая симметричные ветки

В коде две большие половины:

- когда родитель слева;
- когда родитель справа.

Это просто зеркальные случаи.

### Конечный шаг

После выхода из цикла:

```cpp
root_->color = BLACK;
```

Корень обязан быть чёрным.

### Самопроверка

1. Почему вставка сначала идёт как в BST?
2. Почему новый узел создаётся красным?
3. Когда `Insert` возвращает `false`?
4. Что такое дядя в `InsertFixup`?
5. Чем отличается случай "дядя красный" от случая "дядя чёрный"?

### Переходи дальше, если...

Ты можешь словами объяснить три случая вставки: перекраска, треугольник, линия.

## Блок 6. Повороты

### Что понять

Без поворотов `InsertFixup` и `DeleteFixup` не понять.

### `LeftRotate`

Поворот влево вокруг узла `node`:

- его правый ребёнок `pivot` поднимается вверх;
- сам `node` становится левым ребёнком `pivot`;
- поддеревья аккуратно переназначаются.

Ключевая идея:

- относительный порядок ключей не ломается;
- структура дерева меняется;
- свойства BST сохраняются.

### `RightRotate`

Полная зеркальная версия.

### Что поворот НЕ делает

Поворот:

- не меняет сами ключи;
- не меняет значения;
- не делает "перестановку элементов";
- он только переподключает указатели между узлами.

### Самопроверка

1. Что поднимается при левом повороте?
2. Почему после поворота дерево всё ещё остаётся BST?
3. Что именно меняется: значения узлов или связи между ними?

### Переходи дальше, если...

Ты можешь нарисовать левый и правый поворот на 3-4 узлах.

## Блок 7. Удаление — самый тяжёлый кусок

### Что понять

Удаление в RB-дереве сложнее вставки.

Здесь тоже две части:

1. удалить узел как в обычном BST;
2. если надо — восстановить RB-инварианты через `DeleteFixup`.

### Шаг 1. Найти узел

```cpp
Node* target = FindNode(key);
```

Если `target == nil_`, слова нет, возвращаем `false`.

### Шаг 2. Три обычных BST-случая удаления

#### Случай A. Нет левого ребёнка

Тогда вместо узла ставим его правого ребёнка:

```cpp
Transplant(target, target->right);
```

#### Случай B. Нет правого ребёнка

Аналогично, поднимаем левого ребёнка.

#### Случай C. Есть оба ребёнка

Это главный случай.

Тогда:

1. находим `replacement_parent_candidate = Minimum(target->right)`;
2. это следующий по порядку узел;
3. он займёт место удаляемого.

Потом происходят аккуратные перестановки указателей:

- если successor не был непосредственным сыном `target`, его сначала вырезают из старого места;
- затем ставят на место `target`;
- цепляют к нему левое и правое поддеревья `target`;
- цвет successor делают таким же, как у `target`.

### Зачем нужен `original_color`

Перед удалением запоминается цвет физически удаляемого узла:

```cpp
Color original_color = replacement_parent_candidate->color;
```

Почему это важно:

- если удалили красный узел, RB-свойства обычно не ломаются критично;
- если удалили чёрный, могла испортиться чёрная высота, и тогда нужен `DeleteFixup`.

### Что такое `fixup_node`

Это узел, от которого мы начинаем чинить дерево.

Смысл:

- физически удалили один узел;
- "лишняя чёрнота" или проблема баланса как бы остаётся в точке, где произошла замена;
- именно оттуда и стартует `DeleteFixup`.

## Блок 8. `DeleteFixup`

### Что понять

Это самое трудное место всей лабы.

Тут важно понимать не каждую строчку наизусть, а идею 4 случаев.

Цикл идёт, пока:

- `node` не корень;
- `node` чёрный.

То есть мы чиним ситуацию "здесь не хватает чёрной высоты".

### Случай 1. Брат красный

Если брат красный:

- красим брата в чёрный;
- родителя в красный;
- делаем поворот.

Смысл:

- превращаем ситуацию в одну из следующих, где брат уже чёрный.

### Случай 2. Брат чёрный и оба его ребёнка чёрные

Тогда:

- брата красим в красный;
- проблему поднимаем выше, к родителю.

### Случай 3. Брат чёрный, дальний ребёнок чёрный, ближний красный

Тогда делаем поворот у брата.

Смысл:

- превращаем этот случай в следующий, более удобный.

### Случай 4. Брат чёрный и дальний ребёнок красный

Тогда:

- брат получает цвет родителя;
- родитель красится в чёрный;
- дальний ребёнок брата красится в чёрный;
- делается поворот у родителя;
- проблема закрывается.

### Почему код снова разделён на две половины

Потому что:

- один блок — когда `node` левый ребёнок;
- второй — зеркальный случай, когда `node` правый ребёнок.

### Главное, что надо уметь сказать на сдаче

Не надо заучивать код слово в слово.

Надо уметь сказать:

- удаление сначала делается как в BST;
- если физически удалённый узел был чёрным, может нарушиться чёрная высота;
- `DeleteFixup` через перекраски и повороты восстанавливает инварианты;
- случаи делятся по цвету брата и его детей.

### Самопроверка

1. Какие три BST-случая удаления есть до `DeleteFixup`?
2. Зачем хранится `original_color`?
3. Почему `DeleteFixup` нужен не всегда?
4. Какую роль играет брат `sibling`?
5. Что означает "проблема поднимается выше"?

### Переходи дальше, если...

Ты можешь без кода объяснить, что `DeleteFixup` делает в общих чертах и зачем там брат, родитель и повороты.

## Блок 9. Сохранение в файл

### Что понять

Теперь разбираем не дерево, а сериализацию.

### `FileHandle`

Это маленький вспомогательный класс-обёртка над `std::FILE*`.

Он нужен, чтобы файл автоматически закрылся в деструкторе.

То есть это простая форма RAII:

- открыли файл;
- если где-то вышли раньше по ошибке, деструктор всё равно вызовется;
- файл не останется открытым случайно.

### Как работает `Save`

Алгоритм такой:

1. открыть файл на запись;
2. записать сигнатуру файла `RBTD`;
3. записать версию;
4. записать число элементов;
5. обойти дерево in-order и записать каждый элемент.

### Почему обход именно `in-order`

Потому что тогда элементы сохраняются в отсортированном по ключу порядке.

Это удобно:

- формат становится простым;
- при чтении можно проверять уникальность через обычную вставку;
- содержимое файла не зависит от конкретной формы дерева, а зависит от набора слов.

### Что записывается для каждого узла

Для каждого слова пишется:

1. длина слова `uint16_t`;
2. сами байты строки;
3. значение `uint64_t`.

### Формат файла целиком

Файл устроен так:

1. `magic = "RBTD"`
2. `version = 1`
3. `count = количество элементов`
4. `count` раз:
   - `length`
   - `key`
   - `value`

### Почему есть `WriteUint16` и `WriteUint64`

Потому что числа пишутся вручную по байтам.

Это значит:

- формат полностью контролируется нами;
- мы не зависим от внутреннего представления структуры `Node`;
- не пишем в файл "сырую память" объекта.

## Блок 10. Загрузка из файла

### Что понять

`Load` надо понимать очень внимательно, потому что тут важна не только логика чтения, но и защита от битых файлов.

### Как работает `Load`

Алгоритм такой:

1. открыть файл;
2. если файла нет — словарь становится пустым, это считается корректным;
3. иначе читаем всё во временный словарь `loaded_dictionary`;
4. если чтение прошло успешно — делаем `Swap`;
5. если нет — рабочий словарь не меняем.

### Почему чтение идёт во временный словарь

Это очень важная идея.

Если файл:

- битый;
- обрезанный;
- содержит мусор;
- содержит дубликаты;

то текущий словарь не должен испортиться.

Поэтому читаем не в основной объект, а во временный.

Только после полного успеха:

```cpp
Swap(loaded_dictionary);
```

### Что проверяет `ReadFromFile`

1. пустой ли файл;
2. совпадает ли сигнатура;
3. совпадает ли версия;
4. удалось ли прочитать число элементов;
5. корректна ли длина каждого слова;
6. состоят ли символы только из английских букв;
7. нет ли дубликатов при вставке;
8. не осталось ли лишнего мусора в конце файла.

### Почему пустой файл считается валидным

Так требует ТЗ.

Поэтому если файл пуст:

- это не ошибка;
- результат — пустой словарь.

### Откуда берётся `ERROR: invalid file format`

Если структура файла не совпадает с ожидаемой, код создаёт такое сообщение через:

```cpp
MakeInvalidFormatMessage()
```

### Системные ошибки и ошибки формата — это разные вещи

Очень важно различать:

- системные ошибки: нет доступа, ошибка чтения диска, нехватка памяти;
- ошибки формата: файл есть, но внутри не тот формат.

В первом случае:

- сообщение начинается с `ERROR:` и берётся из `strerror`.

Во втором:

- выводится `ERROR: invalid file format`.

## Блок 11. `main`: как программа обрабатывает строки

### Что понять

Теперь можно разобрать внешний слой программы.

### `Trim`

Убирает пробелы и переводы строки по краям.

Это нужно, чтобы:

- пустые строки игнорировались;
- путь после `! Save` / `! Load` корректно читался;
- пользовательский ввод не ломался из-за лишних пробелов по краям.

### Цикл чтения

```cpp
while (std::getline(std::cin, line))
```

То есть программа читает вход построчно до конца файла.

### Ветви обработки

#### 1. Строка начинается с `+`

Разбирается как:

- команда;
- слово;
- число.

Потом вызывается:

```cpp
dictionary.Insert(NormalizeWord(word), value)
```

#### 2. Строка начинается с `-`

Разбирается слово и вызывается `Erase`.

#### 3. Строка начинается с `!`

Читается:

- `Save` или `Load`;
- потом путь.

Дальше вызывается `Save` или `Load`.

#### 4. Иначе

Это просто поиск слова.

### Почему `NormalizeWord` вызывается именно в `main`

Потому что пользователь может ввести слово в любом регистре.

`main` приводит его к нижнему регистру до передачи в словарь.

То есть словарь хранит уже нормализованные ключи.

### Обработка нехватки памяти

Если где-то внутри:

- `new` не смог выделить память;
- или строка не смогла расшириться;

то может вылететь `std::bad_alloc`.

`main` ловит эту ошибку и печатает:

```cpp
ERROR: not enough memory
```

3. Где именно обеспечивается регистронезависимость?

## Блок 12. Полная картина: как всё связано

Теперь собери всё вместе.

### Пример 1. Команда `+ Apple 42`

1. `main` читает строку.
2. `Trim` убирает лишние пробелы.
3. `NormalizeWord("Apple")` даёт `"apple"`.
4. Вызывается `dictionary.Insert("apple", 42)`.
5. Идём по BST-правилу.
6. Создаём красный узел.
7. Чиним дерево через `InsertFixup`.
8. Если всё хорошо — печатаем `OK`.

### Пример 2. Команда `! Save dict.bin`

1. `main` разбирает действие `Save`.
2. Вызывает `dictionary.Save(path, error)`.
3. Открывается файл.
4. Пишется сигнатура, версия, число элементов.
5. Дерево обходится через `WriteInOrder`.
6. Если ошибок нет — печатается `OK`.

### Пример 3. Команда `! Load dict.bin`

1. `main` вызывает `Load`.
2. Открывается файл.
3. Данные читаются во временный словарь.
4. Если формат корректный — временный словарь меняется местами с текущим.
5. Если файл битый — текущий словарь не трогается.

## Блок 13. Что обязательно уметь сказать на сдаче

Вот минимальный набор вещей, которые ты должен уметь объяснить быстро и уверенно.

### 1. Почему это именно красно-чёрное дерево

Потому что:

- это BST;
- у узлов есть цвета;
- после вставки и удаления структура балансируется поворотами и перекрасками;
- высота остаётся логарифмической.

### 2. Зачем нужен `nil_`

Чтобы заменить `nullptr` специальным чёрным узлом и упростить весь код балансировки.

### 3. Как устроена вставка

- сначала как в обычном BST;
- потом `InsertFixup`.

### 4. Как устроено удаление

- сначала как в BST;
- если удалён чёрный узел, вызывается `DeleteFixup`.

### 5. Почему `Load` безопасен

Потому что данные читаются во временный словарь, и текущий словарь меняется только после полного успешного чтения.

### 6. Почему формат бинарный и свой

Потому что по ТЗ нужен компактный бинарный формат, а запись структуры `Node` как есть была бы неправильной и непереносимой.

### 7. Где обеспечивается регистронезависимость

- во входе через `NormalizeWord`;
- при `Load` через проверку и перевод символов в нижний регистр.

### 8. Что делает `FileHandle`

Это маленькая RAII-обёртка, которая закрывает файл автоматически.
